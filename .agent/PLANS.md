# ExecPlan 编写与执行约定

ExecPlan 是用于复杂功能、重要重构和高风险边界工作的可执行计划。它既是实施路线，也是持续
更新的工作记录，使后续 agent 或维护者只依赖当前工作树、计划及其引用的受版本管理文件，
即可理解现状并继续工作，不依赖聊天记录或隐含记忆。

ExecPlan 是非规范性的工程材料。它负责说明如何在一个具体切片中落实项目契约，不定义或
改变产品性质，也不授予或延续操作权限；后续任务仍以用户当前任务消息为授权来源。

## 定位与真相源

不同文件分别承担以下职责：

- `docs/` 定义必须成立的公开行为、安全性质、持久化语义和里程碑边界。
- `README.md`、代码和测试说明当前已经实现的能力。
- `AGENTS.md` 定义仓库级执行、安全和交付规则。
- `CONTRIBUTING.md` 定义分支、提交格式和通用开发流程。
- 本文件定义 ExecPlan 的编写与执行方法。
- 单份 ExecPlan 只记录当前 Goal 的范围、进度、局部决策和验证证据。

存在相关规范时，ExecPlan 必须引用准确文件和章节，并用简短文字说明当前切片依赖的不变量，
但不得复制整套规范形成第二真相源。纯内部重构没有直接对应的产品规范时，应明确声明“不
改变产品契约”，并列出需要保持的 README、代码或测试行为，不得为了形式引用无关章节。

实现与规范冲突时，应修复实现；任务明确要求改变规范时，应先更新相应规范。若规范冲突、
缺失或存在会改变产品性质的未决取舍，暂停实施并请求用户裁决，不得在 ExecPlan 中自行
降低契约。

文档规定“必须成立的性质”，ExecPlan 可以约束实施顺序和验收方法，但除非跨组件契约已经
确定，不应提前规定私有类型、函数名或算法；具体实现由代码和测试决定。

## 何时使用 ExecPlan

以下工作通常应使用 ExecPlan：

- 预计包含多个独立 milestone 或需要跨会话持续推进的任务。
- 涉及路径身份、控制面边界、state、ownership、backup、prune、mutation、持久化格式或
  数据完整性不变量。
- 需要原型研究、平台行为验证、迁移或重要架构调整的工作。
- 用户明确要求使用 ExecPlan 的任务。

小型文字修订、局部测试补充、明确且低风险的 bug 修复通常不需要 ExecPlan，除非它们暴露
了共享不变量或结构性问题。

默认组织方式是：

```text
一个 Goal
  → 一个短生命周期 branch
  → 一份 active ExecPlan
  → 一个或多个可独立验证的 milestone commit
  → 独立复核
  → 人工 review
```

需要改变 Goal 或人工 review 单元时，先请求用户裁决并在 `Decision Log` 中记录。只要不改变
`Scope / Non-goals`，Goal 内部的 milestone 拆分、合并和重排由 agent 自主完成。不得按固定
LOC、文件数或 commit 数机械拆分。

## 存储与生命周期

计划文件的存储、命名和状态迁移以 `.agent/plans/README.md` 为准。ExecPlan 是非规范性的
工作记录，不授予或延续权限；任务授权和 runtime approval 遵循 `AGENTS.md`。

计划定稿后、执行第一个需要授权的操作前，应识别达到 review-ready 所必需的操作，尤其是
branch、stage、commit、计划状态迁移及其失败恢复。必要操作缺少任务授权时，将 `Progress`
标为等待授权，说明操作、对象、影响和必要性，并向用户提出一次最小范围的请求；在获得决定
前暂停依赖该操作的实施，不得改成未提交 diff 或降低验收标准。只有用户明确选择替代交付
方式时，才更新范围与 `Decision Log` 后继续。runtime 边界按 `AGENTS.md` 处理，不改变计划
目标。

默认正式分支流程达到 review-ready，表示当前 Goal 的改动已形成语义 commits、门禁和独立
复核通过，并完成计划状态迁移。任务开始前的无关 dirty 状态只有在 Goal 的 stage 和 commit
能够与其安全隔离，且不会包含、覆盖或改写既有 staged、unstaged 或 untracked 内容时才不
阻止收口；无法安全隔离时，按 `AGENTS.md` 的工作区保护规则请求用户决定。merge、push 和
发布仍是独立操作，不由 Goal 完成状态授权。

## 编写要求

每份 ExecPlan 必须满足以下性质：

- **结果导向**：先说明完成后用户、调用方或测试能够观察到什么。
- **范围明确**：同时写清范围内事项和明确不做的事项。
- **自包含**：不了解历史聊天的新 agent 只依赖当前工作树和计划即可理解完整实施路径，并在
  当前任务授权内继续；计划应显式列出待授权事项，但不能代替用户当前任务消息。受版本管理
  的引用用于导航和取证，不能代替计划所需的假设与解释。
- **可执行**：指出准确的现有路径和符号、预计修改位置、数据流、实施顺序、命令执行目录、
  预期结果以及可辨识的失败特征。
- **可验证**：每个 milestone 都有行为验收，不以“增加了某个类型”代替可观察结果。
- **保持弹性**：规定必须成立的行为，不无依据地锁死内部实现。
- **安全可重试**：说明隔离方式、失败恢复和重复执行是否安全。
- **持续准确**：实际进度、发现和决策变化时立即更新对应章节。
- **叙事优先**：除 `Progress` 外优先使用简洁 prose，按目标、工作、结果和证据组织内容；表格
  或清单只在更清楚时使用。

使用项目内术语时，应在首次出现时简要说明它在本仓库中的含义，并给出相关路径。不要依赖
“之前讨论过”“沿用旧方案”等聊天上下文。

可以引用受版本管理的规范和既有计划，但必须在计划中概括当前切片实际依赖的不变量和上下文，
不得让读者自行拼接关键决策。不得依赖临时参考文件、未纳入仓库的材料或外部网页才能执行。

## Milestone 设计与执行

每个 milestone 必须是可独立解释、验证并形成 checkpoint 的行为切片。先用一段 prose 说明
目标、要完成的工作、结束时新增的结果和证明方式，再给出准确的现有路径或符号、预计修改
位置、测试、从 repo root 或明确目录运行的命令和 commit 边界。存在输入拒绝、安全边界、
mutation 或不可逆风险时，再说明失败特征和安全重试方式。新私有 API 只有在跨组件协调确有
需要时才预先规定。

不得将“完成全部功能并测试”写成单个 milestone。测试与对应实现应在同一语义 commit 中，
不要把全部测试、文档或清理工作拖到最后。特有停止条件只在该 milestone 确有额外边界时写。

执行 milestone 时，先确认工作区和计划进度；适合时增加能暴露缺口的测试，再完成最小完整
实现。窄范围测试、适用门禁、任务 diff 与 untracked 检查、living sections 更新全部完成后，
按 `CONTRIBUTING.md` 创建语义 commit。前一 milestone 的任务改动未提交前，不进入下一项。

执行中发现新的必要操作尚未获得任务授权时，立即更新 `Progress`、提出具体授权请求并暂停
依赖步骤，不得改写 commit 或验收边界继续。默认不 amend、squash 或重写已完成 milestone；
后续问题使用新的 fix commit 并重新运行相关门禁，除非用户明确要求整理历史。

用户当前任务明确要求实施时，可在其范围内编写并执行计划，无需为实现重复请求批准；agent
应在 milestone 之间自主推进。用户明确要求编写或修订计划时，只改计划；纯审查、诊断或
讨论严格只读。只有命中计划停止条件、出现规范取舍、需要扩大范围或缺少必要授权时才暂停。

## Living Plan 维护

以下章节必须持续反映真实状态：

- `Progress`：记录完成、进行中和剩余事项。每个合理停止点都要更新。
- `Surprises & Discoveries`：只记录影响方案、风险或后续工作的意外事实，并附简短证据。
- `Decision Log`：记录会影响行为、范围、架构或验证方式的决定及理由。
- `Outcomes and Handoff`：在收口时对照目标总结结果、偏差、未完成事项和交接状态。

日期使用带年份的 ISO 8601 形式；只有需要分析时长时才记录具体时间。验证证据简要记录命令、
平台和结果，commit 标识只在恢复上下文或 handoff 有帮助时记录。不要粘贴大段日志或易过期
的完整输出。

范围或方案发生实质变化时，应同步更新所有受影响章节。普通进度更新不需要再维护一份重复的
修订日志；Git、`Progress` 和 `Decision Log` 已提供变更轨迹。

## 计划结构

下方模板是计划结构的单一真相源。`Progress`、`Surprises & Discoveries`、`Decision Log` 和
`Outcomes and Handoff` 必须保留；尚无内容时简短写“暂无”即可，不制造带日期的空记录。
可选章节只在确有实质内容时增加。

## 验证、复核与交付

每个 milestone 使用最窄的相关测试快速反馈，并执行 `AGENTS.md` 与 `Makefile` 规定的适用
门禁。具体计划必须写出命令、执行目录、成功判据和可观察验收；只证明能够编译不算完成。
不能在当前环境执行的平台、CI 或外部门禁应明确标为未验证，不得声称通过。

固定门禁、格式和结构校验可以由程序执行，但任务范围、受保护内容、验收充分性及授权状态
仍由 agent 判断。涉及 mutation 时，计划只补充本 Goal 特有的隔离、重试和恢复安排，全局
数据保护边界遵循 `AGENTS.md`。

每个 Goal 收口前必须由未参与实现的只读 subagent 复核全部实质改动。复核上下文应包含项目
定位、Scope / Non-goals、文档与代码分界、相关规范、ExecPlan 和验证结果；高风险计划可以
在实施前增加设计复核。主 agent 负责判断并以新的 fix commit 处理实质问题；修复改变原结论
时重新复核完整改动，否则至少复核新增 fix diff。

当前环境无法提供独立复核或复核尚未完成时，计划必须留在 `active/`，并在 `Progress` 记录
等待独立复核或收口受阻；不得迁移计划、stage 终态收口、创建 plan-closure commit 或声称
review-ready。只有用户明确改变验收边界后，才能更新范围与 `Decision Log` 并按新边界继续。

复核通过后，才按 `.agent/plans/README.md` 执行终态收口：在同一计划中完成 `Progress`、记录
最终门禁和复核结论、更新 `Outcomes and Handoff`、迁移计划、运行 diff check 并创建
plan-closure commit。该 commit 的拟提交 diff 只能包含当前计划的终态更新和迁移；任务开始前
的无关 staged 内容必须保持不变，已确认的未提交计划迁移按目录生命周期规则处理。任务内出现
新的实质改动时，退出机械收口，以新的 fix commit 完成验证和独立复核后再收口。机械收口改动
不触发新一轮独立复核，也不要求计划记录 plan-closure commit 自身的 SHA。

## 停止条件

出现以下情况时，应更新计划并暂停请求裁决或授权：

- 相关规范相互冲突，或必须改变公开行为、安全性质、持久化格式或已接受风险。
- 当前 Goal 无法容纳必要改动，需要扩大范围或拆分新的 Goal。
- Goal 必需的 Git/托管状态变更、真实私人数据或环境、破坏性操作、外部写入或发布操作尚未
  获得任务授权。
- 已获任务授权的必要操作按 `AGENTS.md` 处理 runtime approval 后仍无法执行。
- 对计划外依赖完成有界评估后，新增或替换它仍涉及维护、兼容性、范围或替换成本取舍。
- 无法证明 mutation、恢复或数据完整性边界成立。
- 只能依赖静默 fallback、吞错、跳过测试或近似语义才能继续。

普通实现选择、内部命名和可由代码实验回答的问题不构成停止条件，应自主研究、记录决定并
继续。

## ExecPlan 模板

新计划以以下结构为起点。正文应保持 prose-first；缩进命令、短清单和验收矩阵只用于提高
可执行性。没有实质内容的可选章节应省略。

    # <branch 或 Goal>：<简短、结果导向的标题>

    本 ExecPlan 是 living document。实施期间必须持续更新 `Progress`、
    `Surprises & Discoveries`、`Decision Log` 和
    `Outcomes and Handoff`，并遵循 `.agent/PLANS.md`。

    ## Purpose / Big Picture

    说明这项工作为什么重要。描述完成后用户、调用方或测试能够做到什么，
    以及如何直接观察该结果。

    ## Scope / Non-goals

    范围内：

    - <当前 Goal 必须交付的行为>

    明确不做：

    - <后续里程碑、相邻能力或不属于本分支的内容>

    ## Contract and Context

    - `docs/<file>.md` §<section>：<存在相关规范时，列出当前切片依赖的性质>
    - `docs/09-roadmap.md` §<milestone>：<涉及里程碑能力时，列出当前范围边界>

    用简短 prose 总结当前切片依赖的不变量、基线和缺口，并从不了解实现的维护者
    视角说明准确的现有路径、符号、测试、配置和数据流。引用用于导航和取证，不能
    代替必要解释。纯内部重构应声明“不改变产品契约”，并列出需要保持的 README、
    代码或测试行为；不要复制整份规范或发明新契约。

    ## Progress

    - [x] YYYY-MM-DD：<已完成事项和验证；有助于恢复上下文时记录 commit>
    - [ ] <当前进行中的事项；若部分完成，分别写明已完成与剩余内容>
    - [ ] <后续事项>

    ## Milestones

    ### Milestone 1：<独立可验证的行为切片>

    用一段 prose 按“目标、工作、结果、证据”说明此切片。给出准确的现有路径或
    符号、预计修改位置和数据流；实现细节只写到消除歧义的程度，不提前锁死
    新私有 API。

    Concrete steps：

        在 <repo root 或明确目录> 运行：<准确命令>
        预期：<成功输出或状态；适用时写出可辨识的失败特征>

    验收：

    - <用户、人类 reviewer 或测试可直接观察的成功行为>
    - <适用时写拒绝或失败路径，以及安全重试方式>

    Commit 边界：

        <type(scope): 中文简短摘要>

    仅在确有额外边界时补充本 milestone 的特有停止条件，不重复全局规则。

    存在下一个独立行为切片时，再按相同结构增加 `Milestone 2`；不得用“完成其余功能”
    代替具体结果、测试和验收，也不得为满足固定数量机械拆分。

    ## Validation and Acceptance

    汇总跨 milestone 的最终验收。可以使用紧凑矩阵：

    | 必须成立的性质 | 验证证据 | 状态 |
    |---|---|---|
    | <行为或安全性质> | <测试、命令或人工观察> | 待验证 |

    写出最终命令、执行平台和成功判据。不要写固定测试数量或容易过期的完整输出。

    ## Safety, Authorization, and Recovery

    只记录当前 Goal 特有且达到 review-ready 所必需的受限操作，以及它在当前任务中
    已获授权或仍待授权的证据；不复制 `AGENTS.md` 的通用清单，也不得把状态写成可由
    后续任务继承的权限。本节只保存证据，不产生或延续授权。

    必要授权缺失时，在 `Progress` 标记等待授权，并说明应向用户请求的操作、对象、
    影响和必要性；不得规划未提交 diff 等静默降级。

    说明测试隔离、重复执行、半途失败后的恢复方式，以及如何避免接触真实 HOME、
    repo、config、state、backup 和私人数据。

    ## Interfaces and Dependencies

    仅在涉及跨 package 协作、公开接口、持久化格式或新增依赖时保留本节。

    说明必须形成的接口契约或依赖选择及理由。私有函数签名和算法若不是协调所需，
    留给实现决定。

    ## Surprises & Discoveries

    - Observation: <意外事实>
      Evidence: <简短测试、日志或代码位置>
      Impact: <对方案、范围或风险的影响>

    ## Decision Log

    - Decision: <作出的决定>
      Rationale: <为什么这样选择，以及排除了什么>
      Date: YYYY-MM-DD

    ## Outcomes and Handoff

    对照 `Purpose / Big Picture` 总结实际结果、重要偏差、未完成事项和经验。记录
    base branch、语义 commits、最终门禁、独立复核、已处理意见及未验证项，并说明
    是否满足正式分支流程的 review-ready 条件。

    尚缺必要授权或操作时，明确标为等待授权并给出下一步，不得声称 Goal 已完成。
    merge、push 或发布仍需按任务单独授权。
