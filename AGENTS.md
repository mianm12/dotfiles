# 仓库指南

> [!IMPORTANT]
> `dot` 是单人使用的 dotfiles 管理 CLI，采用 symlink 为主、模板生成为辅的模型。
> 仓库同时承载 CLI 源码和维护者的真实配置；`modules/` 存在后是用户数据，不是测试
> fixture。安全目标是保护数据不被工具自身 bug 损坏，而不是对抗恶意仓库、恶意 hook 或
> 主动并发篡改。
> 目标平台为 macOS 与 Linux，具体完成度以 `docs/09-roadmap.md` 为准；Windows 不在
> 范围内。

## 任务与规范

### 规范来源

- **当前实现**：`README.md`、代码和测试说明已经实现什么。
- **规范契约**：受版本管理的 `docs/` 规定当前及后续里程碑必须成立的性质，不是实现完成
  的证据；代码决定如何实现这些性质。

### 上下文路由

以下规则可以同时适用；命中多项时读取其上下文并集：

- **局部任务**：实现、测试、注释或构建修复先读相关代码、测试与配置；状态不明确时再读
  `README.md`。
- **新能力或里程碑**：读 `README.md`、`docs/09-roadmap.md`，并通过
  `docs/README.md` 定位相关规范。
- **规范相关任务**：涉及公开行为、持久化格式、所有权/覆盖/清理语义、不可逆操作、跨
  组件安全不变量或已接受风险时，先读相关规范；若要改变其中性质，先更新规范。

### 始终适用的规则

- **里程碑范围**：只实现任务覆盖的当前切片，不为后续能力预建；不支持的输入应明确
  拒绝。
- **规范处理**：实现偏离规范时修复代码与测试，不得修改规范迁就旧 bug；规范冲突或契约
  取舍不明确时，指出具体章节并请求裁决。
- **兼容性**：配置使用新版 CLI 能力时，在同一 commit 提升顶层 `requires`。
- **临时材料**：明确标为临时、参考用途或不纳入版本管理的材料不是契约来源，不得在受
  版本管理的内容中引用。

## 自治与安全边界

### 任务内自治与工作区保护

以下规则可以同时适用，更具体、更严格的边界优先。用户已经明确要求具体操作时即构成
授权，不得为同一操作重复询问。

- **任务内自治**：任务范围内的仓库读取、编辑、构建和测试直接执行。
- **只读任务**：仅审查或诊断时，不修改仓库或外部状态。
- **额外确认**：未决规范取舍、真实私人数据或环境写入、破坏性操作、外部写入或扩展任务
  范围，未获任务授权时先确认。
- **工作区保护**：修改前检查 `git status`、相关实现和测试；保留已有 staged、unstaged 和
  untracked 改动，不处理任务外内容。重叠改动无法安全区分时再询问。

### Git、Pull Request 与 Release

- **只读查询**：任何不改变工作区、索引、引用、历史、远端或 GitHub 状态的查询无需额外
  授权，例如 `git status`、`git diff`、`git log` 以及查看 tag、Pull Request 或 Release。
- **状态变更**：任何通过 Git 或托管平台改变上述状态的操作必须由任务明确授权，例如
  创建、切换或删除 branch，stage、commit、amend、rebase、merge、cherry-pick、restore、
  push，创建、应用或删除 stash，以及 tag、Pull Request 或 Release 写操作。

### 真实私人数据

> [!CAUTION]
> 任务未涉及真实 `*.local`、`.env` / `.env.*`、机器配置、state 或 backup 时，不读取也不
> 修改；不得把其中的私密内容带入回复、日志或其他输出。

- **诊断**：任务指定具体路径时，可以做最小必要只读检查。
- **修改或纳入 Git**：须获得针对路径和操作的明确授权。
- **例外**：受版本管理的 example/template 和隔离临时目录中的合成 fixture 不受此限制。

### Mutation 验证隔离

除非用户授权真实环境，手动验证 `apply`、`add`、`init`、`update` 等 mutation 命令必须
同时满足：

- 使用绝对路径的临时 `--home`、repo 和 config。
- 清除或重定向 `DOT_CONFIG`、`DOT_REPO`。
- 让显式 `--repo`、state 和 backup 也指向临时位置。

**仅传 `--home` 不构成完整隔离。**

### 实现安全边界

- **Fail closed**：manifest、state、控制面路径、所有权和不可逆提交等规范安全边界无法
  成立时，在依赖该边界的阶段拒绝相关动作；诊断、恢复及其他规范例外仍按相应规范执行。
- **其他失败**：可以明确报错或按规范降级，不得静默 fallback。
- **威胁边界**：不得超出上述安全模型为恶意环境增加额外防御。

## 仓库与开发

### 仓库地图

以下是核心路径导航，不是完整或固定的架构：

```text
.
├── cmd/dot/main.go           # CLI 入口
├── internal/                 # 实现；*_test.go 与对应代码同目录
├── docs/                     # 规范性契约
├── .github/workflows/ci.yml  # CI 门禁
├── Makefile                  # 常用构建与验证命令
├── README.md                 # 当前实现状态
└── CONTRIBUTING.md           # 分支、提交与开发流程
```

二进制只能写入已忽略的 `bin/`、`dist/` 或仓库外临时目录。

### 常用命令

`Makefile` 和 `make help` 是命令的当前事实来源：

| 命令         | 用途                     | 对仓库的影响                 |
| ------------ | ------------------------ | ---------------------------- |
| `make help`  | 查看当前命令             | 无                           |
| `make build` | 构建 `bin/dot`           | 写入已忽略的构建产物         |
| `make fmt`   | 格式化 Go 代码           | 修改 Go 源码                 |
| `make tidy`  | 整理 Go module           | 可能修改 `go.mod`、`go.sum`  |
| `make check` | 执行与 CI 相同的完整门禁 | 仓库内仅写入已忽略的构建产物 |

### Go 风格

- **工具链**：Go 版本以 `go.mod` 为准，使用 `gofumpt` 和 `goimports`。
- **代码**：优先简短包名、小函数、浅层控制流、`%w` 错误包装和标准接口。
- **注释**：使用中文；标识符、命令、协议、路径及 CLI、HOME、symlink 等技术词保留英文。
  注释解释意图、契约或非显而易见的边界，不复述代码。

### 依赖与抽象

- **依赖**：优先标准库和现有依赖。引入依赖前检查维护状态、采用规模、Go 版本兼容性和
  替换成本，并说明 `go.mod`、`go.sum` 的影响。
- **抽象**：只有真实复用、清晰职责或安全不变量需要单一真相源时才抽象；不得为假想需求
  预建 interface、helper 或层次。

### 测试

- **组织**：使用标准库 `testing`；多个同构场景适合时采用表驱动测试；场景后缀有助于理解
  时使用 `TestSubject_Scenario`。安全路径覆盖成功与拒绝/失败两侧。
- **文件系统**：使用 `t.TempDir` 和真实路径行为。涉及 HOME、repo、config、state、backup
  或 hook 时不得使用或写入真实用户目录，不得绕过生产安全校验；仅为断言其未改变时可做
  最小只读检查。
- **幂等**：涉及 `apply` 收敛行为的测试应验证一次成功收敛后，相同输入再次 `apply` 不产生
  文件 mutation 或重复收养。
- **hook 隔离**：执行 hook 子进程的集成测试必须让其 HOME 和 `XDG_*` 环境全部位于同一
  隔离临时根目录内，并显式断言真实 HOME 未变化。
- **覆盖范围**：不设数字化覆盖率门槛，但应覆盖公开行为、失败路径、顺序不变量和数据保护
  边界。

## 协作与交付

### 协作与独立复核

- **分支与提交**：以 `CONTRIBUTING.md` 为准；执行相关任务前先读该文件。
- **规范复核**：规范主线或跨文档契约发生实质变化后应独立复核；适合时优先使用
  subagent，并提供项目定位、里程碑和文档/代码分界。主 agent 自行确认并处理意见；小型
  文字修订无需强制复核。

### 交付标准

以下规则按触发条件累计适用；命中多项时必须全部完成：

- **任意仓库改动**：检查相对 HEAD 的完整 diff 和任务相关的 untracked 新文件；运行
  `git diff HEAD --check`。
- **本文件维护**：若本任务已确定的改动会使本文件中的长期事实失效，在同一任务中同步修订
  受影响条目；规则仅随用户或规范的明确决定更新，只读任务仅报告。不得机械同步短期状态、
  里程碑进度或具体实现细节。
- **Go 代码、依赖、构建或 CI 改动**：额外运行 `make check`。
- **准备提交**：按 `CONTRIBUTING.md` 执行提交门禁。
- **准备 Pull Request**：说明行为变化、验证命令和文档或兼容性影响；仅在用户可见输出
  变化时附 CLI 示例。
