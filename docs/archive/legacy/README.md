# dot — 个人 dotfiles 管理工具 · 实现前设计基线

> [!WARNING]
> 本目录是重设计前实现及规范的历史存档，已不再具有规范性，也不得用于指导新的实现。
> 当前设计以 [`../../design-baseline.md`](../../design-baseline.md) 为准。

`dot` 是一个单人使用的 dotfiles 管理 CLI,采用「symlink 为主、模板生成为辅」的混合模型。
CLI 源码与配置内容同仓库存放,通过 GitHub Releases 分发二进制,通过 git 同步配置。

本套文档是实现前的**行为与安全基线**,不是实现的逐行蓝图。此后仅当实现改变用户可观察
行为、持久化格式、所有权/覆盖/清理语义、跨组件不变量或已接受风险时更新设计文档;
函数拆分、内部类型、算法细节和测试组织由代码与测试决定。

## 一段话看懂整体设计

仓库根目录同时承载 CLI 和配置;配置集中在 `modules/`,模块目录镜像目标路径,模块顶层
`hooks/` 为保留目录。顶层 manifest 定义 profiles、全局默认和 `requires`,模块 manifest
定义模块自己的 target、文件行为和 hooks。apply 经「加载 → 解析 → 枚举 → 渲染 → 观测
→ 决策 → 校验 → 执行 → 持久化」收敛文件系统。安全内核是:**不自动覆盖或删除未证明
所有权的数据;同一文件系统 target 只有一个逻辑身份;创建先于清理;未收敛时不 prune;
所有不可逆提交发生时相关前提仍须成立;强制替换必须先得到可用备份;工具自己的
repo/config/state/binary 路径永不成为托管目标。**

## 文档与代码的分界

文档规定“必须成立的性质”,代码决定“如何实现这些性质”:

- **规范性**:命令行为、manifest/state 格式、决策表、所有权证据、状态提交时机、锁边界、
  target 身份与路径边界、崩溃恢复结果和明文接受的风险。
- **非规范性**:包名、函数名、内部枚举、临时对象布局、摘要算法与输入的具体编码、target
  身份识别方法、系统调用组合、测试 fixture/编号及故障注入方法。安全关键摘要的**强度**
  仍是规范:偶然碰撞在项目生命周期内必须可忽略;具体算法与编码由实现选定,写入后保持
  可识别、可迁移。
- **性质与手段的边界**:例如“提交时仍基于同一对象和证据”是规范;通过提交前复核、
  不覆盖创建原语、目录句柄或其他机制满足它是实现选择。文中的 pipeline 和动作名用于
  表达阶段边界与可观察结果,不把某一种系统调用时序固化为唯一实现。
- **裁决顺序**:威胁模型与安全不变量 > 用户可观察行为 > 持久化兼容 > 示例与实现建议。

测试是规范的可执行验证,不是另一个设计来源。实现中发现新问题时,先判断它是否改变上述
规范性内容;若不改变,直接在代码和测试中解决,不扩写本文档。
